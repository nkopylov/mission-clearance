Architecture

Crate map, data flow, policy pipeline, trace system, and vault internals.

System Overview

Mission Clearance is structured as a Rust workspace with 10 focused crates. The plugin layer (Python scripts) handles integration with Claude Code, while the Rust binary provides the kernel, policy engine, trace log, vault, permission graph, and identity resolution.

Crate Map

mc-core

Domain types: missions, capabilities, operations, policies, resources. The shared vocabulary used by all other crates.

mc-kernel

Classifier, capability checker, content analyzer, and session tracker. The brain that classifies every operation.

mc-policy

Deterministic policy pipeline with escalation support. Evaluates rules and produces Allow, Deny, or Escalate verdicts.

mc-trace

Append-only SHA-256 chained event log and mission delegation graph. Provides tamper-evident audit trails.

mc-vault

AES-256-GCM encrypted credential store with Argon2id key derivation. Secrets never leave the vault unencrypted.

mc-api

Internal HTTP server built with axum. Used by the hook and MCP tools to communicate with the kernel.

mc-sdk

Embedded kernel used internally by the server. Provides a Rust API for kernel operations.

mc-cli

Binary entry point. Runs the server with mission-clearance start. Handles CLI argument parsing and startup.

mc-graph

Permission graph with SQLite-backed storage. Manages principals, roles, RBAC, delegation edges, chain verification, bounded authorizations, and cascading revocation.

mc-identity

Identity resolution via external providers. OIDC/JWT token validation, API key mapping, mission token resolution, and SCIM org chart synchronization.

Permission Graph

The permission graph adds identity, RBAC, and delegation verification on top of the existing capability kernel. When configured, it answers: who authorized this agent, can they authorize this, and is the entire delegation chain still valid?

Delegation Graph (Layer 3)

Can Person B delegate capability C to Agent Z via Agent A? Principals are connected by delegation edges with scoped constraints (time bounds, operation limits, resource narrowing, sub-delegation control). Chain verification walks from leaf to root, validating every link.

RBAC (Layer 2)

Does Person B have capability C at all? Role assignments with hierarchical inheritance, composite roles, scoped assignments (global, team, resource), and separation of duties (conflicting roles cannot be held simultaneously).

Capability Kernel (Layer 1)

Existing fast-path check, unchanged. Does this mission have capability C for resource R? Layers 2 and 3 run in parallel and feed context (trust level, chain anomaly flags) into the policy pipeline.

Cascading Revocation

Revoking a principal or delegation edge cascades through the graph. All downstream agents and their missions are recursively revoked. Supports partial revocation of specific capabilities or edges.

Data Flow

Every tool call follows this path from Claude Code to a policy decision.

1

Tool call intercepted

Claude Code triggers a tool call (Bash, Edit, Write, etc.). The PreToolUse hook receives the raw tool name and input parameters.

2

Context expansion

The hook enriches the operation: it reads script contents for Bash commands, expands package.json lifecycle hooks for npm install, resolves cross-file function bodies for imports, and expands Makefile targets.

3

HTTP request to MC server

The enriched operation is sent to the MC server at 127.0.0.1:9090 via HTTP POST.

4

Kernel classification

The kernel classifies the operation along multiple dimensions: resource type (file, shell, network, etc.), destructiveness (None through Catastrophic), reversibility, data flow direction, and pattern matching against known attacks.

5

Policy evaluation

The classification feeds into the policy pipeline. Deterministic rules are evaluated in priority order. The first matching rule determines the verdict.

6

Trace recording

The operation, classification, and decision are appended to the SHA-256 chained trace log. Anomalies (taint, drift) are recorded separately.

Policy Pipeline

The policy pipeline is a layered evaluation system. The deterministic layer runs in-process and handles clear-cut cases. When it returns "Escalate," the integration layer asks the host LLM (Claude) to judge the operation.

Trace System

The trace system provides a tamper-evident audit log of every operation and decision.

SHA-256 Hash Chaining

Each trace event includes the hash of the previous event, forming a chain. Any modification to a past event breaks the chain, making tampering detectable. Integrity is verified with chain_integrity = true.

Event Types

The trace records: operation submissions, classifications, policy decisions (allow/deny/escalate), escalation outcomes, mission lifecycle events (create/delegate/revoke), and anomaly detections.

Delegation Graph

Missions form a tree via delegation. The trace maintains a graph of parent-child relationships, exportable as DOT format for visualization. Revocation cascades down the tree.

Anomaly Detection

The trace tracks three categories of anomalies: taint propagation (write-then-execute chains), goal drift (operations unrelated to the mission), and prompt injection attempts.

Vault System

The vault provides encrypted credential storage so that secrets do not need to be stored in plaintext in environment variables or config files.

Encryption

Credentials are encrypted with AES-256-GCM, an authenticated encryption scheme that provides both confidentiality and integrity. Each credential has a unique nonce.

Key Derivation

The encryption key is derived from a passphrase using Argon2id, a memory-hard key derivation function resistant to GPU and ASIC attacks. The passphrase is provided via MC_VAULT_PASSPHRASE. 

# Vault operations
/mission-clearance:vault add      # Store a credential (encrypted at rest)
/mission-clearance:vault list     # List credentials (metadata only, no secrets)
/mission-clearance:vault rotate   # Update a credential's value
/mission-clearance:vault revoke   # Delete a credential
