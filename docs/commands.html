Commands & MCP Tools

Slash commands for humans, MCP tools for Claude.

How the Hook/MCP Architecture Works

Mission Clearance integrates with Claude Code through two mechanisms:

PreToolUse Hook

A Python script (hooks/pre-tool-use.py) registered via hooks.json. It intercepts every tool call Claude makes, enriches it with context (script contents, package lifecycle hooks, cross-file function bodies), and sends it to the MC server for policy evaluation. Runs transparently — no user action needed.

MCP Server

A stdio JSON-RPC server (scripts/mc-mcp-server.py) that exposes tools Claude can call directly. These tools let Claude manage missions, handle escalations (approve/deny), query traces, and manage vault credentials. Configured via .mcp.json.

Slash Commands

Markdown-based command definitions in commands/*.md. These provide the user-facing interface: /mission-clearance:start, :stop, :status, etc. Each command instructs Claude to invoke the appropriate MCP tools.

Slash Commands

These commands are typed by the user in the Claude Code chat. Each one instructs Claude to perform a specific session management action.

Command Description
/mission-clearance:start [goal] Launch the MC server (if not already running), create a root mission scoped to the given goal, and activate gating. Saves session state to .claude/mc-session.json.
/mission-clearance:stop Revoke the active mission and deactivate gating. The MC server continues running but no longer blocks tool calls.
/mission-clearance:status Show the active mission, server health, recent decisions, and any detected anomalies.
/mission-clearance:trace [mode] View trace data. Modes: events (all events), denials (blocked operations), anomalies (detected anomalies), graph (mission delegation graph).
/mission-clearance:vault <action> Manage encrypted credentials. Actions: add (store a credential), list (show metadata), rotate (update a credential), revoke (delete a credential).
/mission-clearance:delegate <goal> Create a sub-mission with narrowed permissions, delegated from the current active mission.

MCP Tools

These tools are exposed via the MCP server and called by Claude during sessions. They form the programmatic API for mission management, policy evaluation, and escalation handling.

Tool Description
mc_create_mission Create a root mission with a goal and capabilities. Returns a mission ID used for all subsequent operations.
mc_get_mission Get details of a mission by ID, including its status, capabilities, and child missions.
mc_delegate_mission Delegate a sub-mission with narrowed permissions. The child mission cannot exceed the parent's capabilities.
mc_revoke_mission Revoke a mission and all its sub-missions. Revocation cascades down the delegation tree.
mc_submit_operation Submit an operation for policy evaluation. Returns Allow, Deny, or Escalate with an explanation.
mc_vault_add Add a credential to the encrypted vault (AES-256-GCM with Argon2id key derivation).
mc_vault_list List credentials stored in the vault. Returns metadata only — secrets are never exposed.
mc_vault_revoke Revoke (delete) a vault credential by name.
mc_trace_events Get trace events, optionally filtered by mission ID. Returns the append-only event log.
mc_trace_anomalies Get detected anomalies: taint propagation, goal drift, prompt injection attempts.
mc_policy_test Dry-run a policy evaluation without a mission context. Useful for testing rules.
mc_server_status Check if the MC server is running and healthy. Returns version, uptime, and active mission count.
mc_approve_escalation LLM Judge: approve an escalated operation. The approval is time-limited (5 minutes) and single-use.
mc_deny_escalation LLM Judge: deny an escalated operation with a reason.

Escalation Flow

When the policy pipeline returns "Escalate," the hook pauses execution and Claude itself acts as the judge. Here is the flow:

1

Policy returns Escalate

The deterministic layer cannot make a clear Allow/Deny decision. The operation is flagged for review.

2

Claude reviews the operation

Claude receives the operation details, classification, and the reason for escalation. It evaluates the context and intent.

3

Claude calls approve or deny

Claude invokes mc_approve_escalation or mc_deny_escalation with a reason. Approvals are time-limited to 5 minutes and single-use.

4

Decision is recorded

The escalation outcome is logged in the trace with the judge's reasoning, creating an audit trail.
